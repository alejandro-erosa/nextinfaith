## 5. Deuda técnica — estado completo

Esta sección documenta toda la deuda técnica conocida del proyecto. Es crítica para cualquier transferencia o salida del proyecto.

| # | Deuda | Descripción | Riesgo | Condición de salida |
|---|---|---|---|---|
| 1 | **Capa de servicios BD** | Los componentes llaman directamente a Supabase sin abstracción. Si se cambia de proveedor de BD, hay que tocar cada componente | ALTO — migración costosa | Implementar `app/services/` con funciones que encapsulen todas las queries. Todo código nuevo debe usar servicios |
| 2 | **Campos TEXT deprecados en `eventos`** | `ciudad`, `estado`, `pais`, `modalidad`, `estado_publicacion` aún existen como TEXT aunque fueron reemplazados por FKs | MEDIO — datos inconsistentes si alguien escribe en el campo viejo | Ejecutar DROP COLUMN cuando se confirme que ningún código los referencia |
| 3 | **Normalización pendiente en otras tablas** | `profiles`, `campanas_publicitarias`, `places`: `ciudad_id` existe en BD pero el código no lo usa | MEDIO — inconsistencia de datos | Actualizar código para usar `ciudad_id` en lugar de TEXT libre |
| 4 | **Confirmar email desactivado** | Supabase Auth tiene `confirm email = OFF`. Cualquier email puede registrarse sin verificación | ALTO — riesgo de spam y cuentas falsas | Reactivar antes de onboarding de usuarios externos |
| 5 | **RLS desactivado en algunas tablas** | `actividad_log` y tablas ext_* tienen RLS desactivado temporalmente | ALTO — datos expuestos | Reactivar antes de usuarios externos |
| 6 | **Video hero sin solución definitiva** | El video fue revertido a imagen estática. La experiencia visual original era mejor | BAJO — solo estético | Implementar YouTube embed sin controles para servir el video sin consumir bandwidth propio |
| 7 | **Supabase Auth no abstraído completamente** | `UserContext` centraliza el 90% pero `auth/recovery` y `auth/forgot-password` llaman a Supabase directamente | BAJO — solo afecta migración futura de Auth | Mover las llamadas restantes a helpers en UserContext cuando se refactorice Auth |
| 8 | **Datos fake en BD** | Eventos y perfiles de prueba mezclados con datos reales | ALTO — confunde a usuarios externos | Ejecutar DELETE de registros fake antes de V1.0 |

## 6. Principios de arquitectura establecidos

Estos principios aplican a todo código nuevo y son requisito para una transferencia limpia del proyecto:

1. **Independencia de proyectos** — NIF y Toparme tienen BDs, cuentas de hosting y repos completamente separados
2. **Normalización obligatoria** — ningún campo de texto libre para datos estructurados. Siempre catálogos con FK
3. **Auth centralizado en UserContext** — ningún componente llama a `supabase.auth` directamente
4. **Sin warnings en commits** — `npm run build` limpio antes de cada push
5. **Deuda técnica documentada** — cualquier decisión temporal se registra con su razón y condición de salida
6. **Capa de servicios en código nuevo** — todo código nuevo usa `app/services/` en lugar de queries directas
